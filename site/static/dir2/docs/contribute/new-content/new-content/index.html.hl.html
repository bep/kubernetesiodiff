  Opening a pull request
</h1>
<p>

<h1>
  Opening a pull request
</h1>
<blockquote class="note">
  <div>
    <strong>
      Note:
    </strong>
    <strong>
      Code developers
    </strong>
    : If you are documenting a new feature for an
    upcoming Kubernetes release, see
    <a href="/docs/contribute/new-content/new-features/">
      Document a new feature
    </a>
    .
  </div>
</blockquote>
<p>
  To contribute new content pages or improve existing content pages, open a pull request (PR). Make sure you follow all the requirements in the
  <a href="/docs/contribute/new-content/overview/#before-you-begin">
    Before you begin
  </a>
  section.
</p>
<p>
  If your change is small, or you&rsquo;re unfamiliar with git, read
  <a href="#changes-using-github">
    Changes using GitHub
  </a>
  to learn how to edit a page.
</p>
<p>
  If your changes are large, read
  <a href="#fork-the-repo">
    Work from a local fork
  </a>
  to learn how to make changes locally on your computer.
</p>

<h2 id="changes-using-github">
  Changes using GitHub
</h2>
<p>
  If you&rsquo;re less experienced with git workflows, here&rsquo;s an easier method of
  opening a pull request.
</p>
<ol>
  <li>
    <p>
      On the page where you see the issue, select the pencil icon at the top right.
      You can also scroll to the bottom of the page and select
      <strong>
        Edit this page
      </strong>
      .
    </p>
  </li>
  <li>
    <p>
      Make your changes in the GitHub markdown editor.
    </p>
  </li>
  <li>
    <p>
      Below the editor, fill in the
      <strong>
        Propose file change
      </strong>
      form. In the first field, give your commit message a title. In
      the second field, provide a description.
    </p>
    <blockquote class="note">
  </li>
</ol>
<div>
  <strong>
    Note:
  </strong>
  Do not use any
  <a href="https://help.github.com/en/github/managing-your-work-on-github/linking-a-pull-request-to-an-issue#linking-a-pull-request-to-an-issue-using-a-keyword">
    GitHub Keywords
  </a>
  in your commit message. You can add those to the pull request
  description later.
</div>
<ol start="4">
  <li>
    <p>
      Select
      <strong>
        Propose file change
      </strong>
      .
    </p>
  </li>
  <li>
    <p>
      Select
      <strong>
        Create pull request
      </strong>
      .
    </p>
  </li>
  <li>
    <p>
      The
      <strong>
        Open a pull request
      </strong>
      screen appears. Fill in the form:
    </p>
    <ul>
      <li>
        The
        <strong>
          Subject
        </strong>
        field of the pull request defaults to the commit summary.
        You can change it if needed.
      </li>
      <li>
        The
        <strong>
          Body
        </strong>
        contains your extended commit message, if you have one,
        and some template text. Add the
        details the template text asks for, then delete the extra template text.
      </li>
      <li>
        Leave the
        <strong>
          Allow edits from maintainers
        </strong>
        checkbox selected.
      </li>
    </ul>
    <blockquote class="note">
  </li>
</ol>
<div>
  <strong>
    Note:
  </strong>
  PR descriptions are a great way to help reviewers understand your change. For more information, see
  <a href="#open-a-pr">
    Opening a PR
  </a>
  .
</div>
<ol start="7">
  <li>
    Select
    <strong>
      Create pull request
    </strong>
    .
  </li>
</ol>
<h3 id="addressing-feedback-in-github">
  Addressing feedback in GitHub
</h3>
<p>
  Before merging a pull request, Kubernetes community members review and
  approve it. The
  <code>
    k8s-ci-robot
  </code>
  suggests reviewers based on the nearest
  owner mentioned in the pages. If you have someone specific in mind,
  leave a comment with their GitHub username in it.
</p>
<p>
  If a reviewer asks you to make changes:
</p>
<ol>
  <li>
    Go to the
    <strong>
      Files changed
    </strong>
    tab.
  </li>
  <li>
    Select the pencil (edit) icon on any files changed by the
    pull request.
  </li>
  <li>
    Make the changes requested.
  </li>
  <li>
    Commit the changes.
  </li>
</ol>
<p>
  If you are waiting on a reviewer, reach out once every 7 days. You can also post a message in the
  <code>
    #sig-docs
  </code>
  Slack channel.
</p>
<p>
  When your review is complete, a reviewer merges your PR and your changes go live a few minutes later.
</p>
<h2 id="fork-the-repo">
  Work from a local fork
</h2>
<p>
  If you&rsquo;re more experienced with git, or if your changes are larger than a few lines,
  work from a local fork.
</p>
<p>
  Make sure you have
  <a href="https://git-scm.com/book/en/v2/Getting-Started-Installing-Git">
    git
  </a>
  installed on your computer. You can also use a git UI application.
</p>
<h3 id="fork-the-kubernetes-website-repository">
  Fork the kubernetes/website repository
</h3>
<ol>
  <li>
    Navigate to the
    <a href="https://github.com/kubernetes/website/">
      <code>
        kubernetes/website
      </code>
    </a>
    repository.
  </li>
  <li>
    Select
    <strong>
      Fork
    </strong>
    .
  </li>
</ol>
<h3 id="create-a-local-clone-and-set-the-upstream">
  Create a local clone and set the upstream
</h3>
<ol start="3">
  <li>
    <p>
      In a terminal window, clone your fork:
    </p>
    <div class="highlight">
      <pre style="background-color:#f8f8f8;-moz-tab-size:4;-o-tab-size:4;tab-size:4"><code class="language-bash" data-lang="bash">git clone git@github.com/&lt;github_username&gt;/website</code></pre>
    </div>
  </li>
  <li>
    <p>
      Navigate to the new
      <code>
        website
      </code>
      directory. Set the
      <code>
        kubernetes/website
      </code>
      repository as the
      <code>
        upstream
      </code>
      remote:
    </p>
    <div class="highlight">
      <pre style="background-color:#f8f8f8;-moz-tab-size:4;-o-tab-size:4;tab-size:4"><code class="language-bash" data-lang="bash"><span style="color:#a2f">cd</span> website

git remote add upstream https://github.com/kubernetes/website.git</code></pre>
    </div>
  </li>
  <li>
    <p>
      Confirm your
      <code>
        origin
      </code>
      and
      <code>
        upstream
      </code>
      repositories:
    </p>
    <div class="highlight">
      <pre style="background-color:#f8f8f8;-moz-tab-size:4;-o-tab-size:4;tab-size:4"><code class="language-bash" data-lang="bash">git remote -v</code></pre>
    </div>
    <p>
      Output is similar to:
    </p>
    <div class="highlight">
      <pre style="background-color:#f8f8f8;-moz-tab-size:4;-o-tab-size:4;tab-size:4"><code class="language-bash" data-lang="bash">origin	git@github.com:&lt;github_username&gt;/website.git <span style="color:#666">(</span>fetch<span style="color:#666">)</span>
origin	git@github.com:&lt;github_username&gt;/website.git <span style="color:#666">(</span>push<span style="color:#666">)</span>
upstream	https://github.com/kubernetes/website <span style="color:#666">(</span>fetch<span style="color:#666">)</span>
upstream	https://github.com/kubernetes/website <span style="color:#666">(</span>push<span style="color:#666">)</span></code></pre>
    </div>
  </li>
  <li>
    <p>
      Fetch commits from your fork&rsquo;s
      <code>
        origin/master
      </code>
      and
      <code>
        kubernetes/website
      </code>
      's
      <code>
        upstream/master
      </code>
      :
    </p>
    <div class="highlight">
      <pre style="background-color:#f8f8f8;-moz-tab-size:4;-o-tab-size:4;tab-size:4"><code class="language-bash" data-lang="bash">git fetch origin
git fetch upstream</code></pre>
    </div>
    <p>
      This makes sure your local repository is up to date before you start making changes.
    </p>
    <blockquote class="note">
  </li>
</ol>
<div>
  <strong>
    Note:
  </strong>
  This workflow is different than the
  <a href="https://github.com/kubernetes/community/blob/master/contributors/guide/github-workflow.md">
    Kubernetes Community GitHub Workflow
  </a>
  . You do not need to merge your local copy of
  <code>
    master
  </code>
  with
  <code>
    upstream/master
  </code>
  before pushing updates to your fork.
</div>
<h3 id="create-a-branch">
  Create a branch
</h3>
<ol>
  <li>
    Decide which branch base to your work on:
  </li>
</ol>
<ul>
  <li>
    <p>
      For improvements to existing content, use
      <code>
        upstream/master
      </code>
      .
    </p>
  </li>
  <li>
    <p>
      For new content about existing features, use
      <code>
        upstream/master
      </code>
      .
    </p>
  </li>
  <li>
    <p>
      For localized content, use the localization&rsquo;s conventions. For more information, see
      <a href="/docs/contribute/localization/">
        localizing Kubernetes documentation
      </a>
      .
    </p>
  </li>
  <li>
    <p>
      For new features in an upcoming Kubernetes release, use the feature branch. For more information, see
      <a href="/docs/contribute/new-content/new-features/">
        documenting for a release
      </a>
      .
    </p>
  </li>
  <li>
    <p>
      For long-running efforts that multiple SIG Docs contributors collaborate on,
      like content reorganization, use a specific feature branch created for that
      effort.
    </p>
    <p>
      If you need help choosing a branch, ask in the
      <code>
        #sig-docs
      </code>
      Slack channel.
    </p>
  </li>
</ul>
<ol start="2">
  <li>
    <p>
      Create a new branch based on the branch identified in step 1. This example assumes the base branch is
      <code>
        upstream/master
      </code>
      :
    </p>
    <div class="highlight">
      <pre style="background-color:#f8f8f8;-moz-tab-size:4;-o-tab-size:4;tab-size:4"><code class="language-bash" data-lang="bash">git checkout -b &lt;my_new_branch&gt; upstream/master</code></pre>
    </div>
  </li>
  <li>
    <p>
      Make your changes using a text editor.
    </p>
  </li>
</ol>
<p>
  At any time, use the
  <code>
    git status
  </code>
  command to see what files you&rsquo;ve changed.
</p>
<h3 id="commit-your-changes">
  Commit your changes
</h3>
<p>
  When you are ready to submit a pull request, commit your changes.
</p>
<ol>
  <li>
    <p>
      In your local repository, check which files you need to commit:
    </p>
    <div class="highlight">
      <pre style="background-color:#f8f8f8;-moz-tab-size:4;-o-tab-size:4;tab-size:4"><code class="language-bash" data-lang="bash">git status</code></pre>
    </div>
    <p>
      Output is similar to:
    </p>
    <div class="highlight">
      <pre style="background-color:#f8f8f8;-moz-tab-size:4;-o-tab-size:4;tab-size:4"><code class="language-bash" data-lang="bash">On branch &lt;my_new_branch&gt;
Your branch is up to date with <span style="color:#b44">&#39;origin/&lt;my_new_branch&gt;&#39;</span>.

Changes not staged <span style="color:#a2f;font-weight:bold">for</span> commit:
<span style="color:#666">(</span>use <span style="color:#b44">&#34;git add &lt;file&gt;...&#34;</span> to update what will be committed<span style="color:#666">)</span>
<span style="color:#666">(</span>use <span style="color:#b44">&#34;git checkout -- &lt;file&gt;...&#34;</span> to discard changes in working directory<span style="color:#666">)</span>

modified:   content/en/docs/contribute/new-content/contributing-content.md

no changes added to commit <span style="color:#666">(</span>use <span style="color:#b44">&#34;git add&#34;</span> and/or <span style="color:#b44">&#34;git commit -a&#34;</span><span style="color:#666">)</span></code></pre>
    </div>
  </li>
  <li>
    <p>
      Add the files listed under
      <strong>
        Changes not staged for commit
      </strong>
      to the commit:
    </p>
    <div class="highlight">
      <pre style="background-color:#f8f8f8;-moz-tab-size:4;-o-tab-size:4;tab-size:4"><code class="language-bash" data-lang="bash">git add &lt;your_file_name&gt;</code></pre>
    </div>
    <p>
      Repeat this for each file.
    </p>
  </li>
  <li>
    <p>
      After adding all the files, create a commit:
    </p>
    <div class="highlight">
      <pre style="background-color:#f8f8f8;-moz-tab-size:4;-o-tab-size:4;tab-size:4"><code class="language-bash" data-lang="bash">git commit -m <span style="color:#b44">&#34;Your commit message&#34;</span></code></pre>
    </div>
    <blockquote class="note">
  </li>
</ol>
<div>
  <strong>
    Note:
  </strong>
  Do not use any
  <a href="https://help.github.com/en/github/managing-your-work-on-github/linking-a-pull-request-to-an-issue#linking-a-pull-request-to-an-issue-using-a-keyword">
    GitHub Keywords
  </a>
  in your commit message. You can add those to the pull request
  description later.
</div>
<ol start="4">
  <li>
    <p>
      Push your local branch and its new commit to your remote fork:
    </p>
    <div class="highlight">
      <pre style="background-color:#f8f8f8;-moz-tab-size:4;-o-tab-size:4;tab-size:4"><code class="language-bash" data-lang="bash">git push origin &lt;my_new_branch&gt;</code></pre>
    </div>
  </li>
</ol>

<h3 id="addressing-feedback-locally">
  Addressing feedback locally
</h3>
<ol>
  <li>
    <p>
      After making your changes, amend your previous commit:
    </p>
    <div class="highlight">
      <pre style="background-color:#f8f8f8;-moz-tab-size:4;-o-tab-size:4;tab-size:4"><code class="language-bash" data-lang="bash">git commit -a --amend</code></pre>
    </div>
    <ul>
      <li>
        <code>
          -a
        </code>
        : commits all changes
      </li>
      <li>
        <code>
          --amend
        </code>
        : amends the previous commit, rather than creating a new one
      </li>
    </ul>
  </li>
  <li>
    <p>
      Update your commit message if needed.
    </p>
  </li>
  <li>
    <p>
      Use
      <code>
        git push origin &lt;my_new_branch&gt;
      </code>
      to push your changes and re-run the Netlify tests.
    </p>

  </li>

  <strong>
    Note:
  </strong>
  If you use
  <code>
    git commit -m
  </code>
  instead of amending, you must
  <a href="#squashing-commits">
    squash your commits
  </a>
  before merging.

<h4 id="changes-from-reviewers">
  Changes from reviewers
</h4>
<p>
  Sometimes reviewers commit to your pull request. Before making any other changes, fetch those commits.
</p>
<ol>
  <li>
    <p>
      Fetch commits from your remote fork and rebase your working branch:
    </p>
    <div class="highlight">
      <pre style="background-color:#f8f8f8;-moz-tab-size:4;-o-tab-size:4;tab-size:4"><code class="language-bash" data-lang="bash">git fetch origin
git rebase origin/&lt;your-branch-name&gt;</code></pre>
    </div>
  </li>
  <li>
    <p>
      After rebasing, force-push new changes to your fork:
    </p>
    <div class="highlight">
      <pre style="background-color:#f8f8f8;-moz-tab-size:4;-o-tab-size:4;tab-size:4"><code class="language-bash" data-lang="bash">git push --force-with-lease origin &lt;your-branch-name&gt;</code></pre>
    </div>
  </li>
</ol>
<h4 id="merge-conflicts-and-rebasing">
  Merge conflicts and rebasing
</h4>
<blockquote class="note">
  <div>
    <strong>
      Note:
    </strong>
    For more information, see
    <a href="https://git-scm.com/book/en/v2/Git-Branching-Basic-Branching-and-Merging#_basic_merge_conflicts">
      Git Branching - Basic Branching and Merging
    </a>
    ,
    <a href="https://git-scm.com/book/en/v2/Git-Tools-Advanced-Merging">
      Advanced Merging
    </a>
    , or ask in the
    <code>
      #sig-docs
    </code>
    Slack channel for help.
  </div>
</blockquote>
<p>
  If another contributor commits changes to the same file in another PR, it can create a merge conflict. You must resolve all merge conflicts in your PR.
</p>
<ol>
  <li>
    <p>
      Update your fork and rebase your local branch:
    </p>
    <div class="highlight">
      <pre style="background-color:#f8f8f8;-moz-tab-size:4;-o-tab-size:4;tab-size:4"><code class="language-bash" data-lang="bash">git fetch origin
git rebase origin/&lt;your-branch-name&gt;</code></pre>
    </div>
    <p>
      Then force-push the changes to your fork:
    </p>
    <div class="highlight">
      <pre style="background-color:#f8f8f8;-moz-tab-size:4;-o-tab-size:4;tab-size:4"><code class="language-bash" data-lang="bash">git push --force-with-lease origin &lt;your-branch-name&gt;</code></pre>
    </div>
  </li>
  <li>
    <p>
      Fetch changes from
      <code>
        kubernetes/website
      </code>
      's
      <code>
        upstream/master
      </code>
      and rebase your branch:
    </p>
    <div class="highlight">
      <pre style="background-color:#f8f8f8;-moz-tab-size:4;-o-tab-size:4;tab-size:4"><code class="language-bash" data-lang="bash">git fetch upstream
git rebase upstream/master</code></pre>
    </div>
  </li>
  <li>
    <p>
      Inspect the results of the rebase:
    </p>
    <div class="highlight">
      <pre style="background-color:#f8f8f8;-moz-tab-size:4;-o-tab-size:4;tab-size:4"><code class="language-bash" data-lang="bash">git status</code></pre>
    </div>
  </li>
</ol>
<p>
  This results in a number of files marked as conflicted.
</p>
<ol start="4">
  <li>
    <p>
      Open each conflicted file and look for the conflict markers:
      <code>
        &gt;&gt;&gt;
      </code>
      ,
      <code>
        &lt;&lt;&lt;
      </code>
      , and
      <code>
        ===
      </code>
      . Resolve the conflict and delete the conflict marker.
    </p>
    <blockquote class="note">
  </li>
</ol>
<div>
  <strong>
    Note:
  </strong>
  For more information, see
  <a href="https://git-scm.com/docs/git-merge#_how_conflicts_are_presented">
    How conflicts are presented
  </a>
  .
</div>
<ol start="5">
  <li>
    <p>
      Add the files to the changeset:
    </p>
    <div class="highlight">
      <pre style="background-color:#f8f8f8;-moz-tab-size:4;-o-tab-size:4;tab-size:4"><code class="language-bash" data-lang="bash">git add &lt;filename&gt;</code></pre>
    </div>
  </li>
  <li>
    <p>
      Continue the rebase:
    </p>
    <div class="highlight">
      <pre style="background-color:#f8f8f8;-moz-tab-size:4;-o-tab-size:4;tab-size:4"><code class="language-bash" data-lang="bash">git rebase --continue</code></pre>
    </div>
  </li>
  <li>
    <p>
      Repeat steps 2 to 5 as needed.
    </p>
    <p>
      After applying all commits, the
      <code>
        git status
      </code>
      command shows that the rebase is complete.
    </p>
  </li>
  <li>
    <p>
      Force-push the branch to your fork:
    </p>
    <div class="highlight">
      <pre style="background-color:#f8f8f8;-moz-tab-size:4;-o-tab-size:4;tab-size:4"><code class="language-bash" data-lang="bash">git push --force-with-lease origin &lt;your-branch-name&gt;</code></pre>
    </div>
    <p>
      The pull request no longer shows any conflicts.
    </p>
  </li>
</ol>
<h3 id="squashing-commits">
  Squashing commits
</h3>
<blockquote class="note">
  <div>
    <strong>
      Note:
    </strong>
    For more information, see
    <a href="https://git-scm.com/book/en/v2/Git-Tools-Rewriting-History">
      Git Tools - Rewriting History
    </a>
    , or ask in the
    <code>
      #sig-docs
    </code>
    Slack channel for help.
  </div>
</blockquote>
<p>
  If your PR has multiple commits, you must squash them into a single commit before merging your PR. You can check the number of commits on your PR&rsquo;s
  <strong>
    Commits
  </strong>
  tab or by running the
  <code>
    git log
  </code>
  command locally.
</p>
<blockquote class="note">
  <div>
    <strong>
      Note:
    </strong>
    This topic assumes
    <code>
      vim
    </code>
    as the command line text editor.
  </div>
</blockquote>
<ol>
  <li>
    <p>
      Start an interactive rebase:
    </p>
    <div class="highlight">
      <pre style="background-color:#f8f8f8;-moz-tab-size:4;-o-tab-size:4;tab-size:4"><code class="language-bash" data-lang="bash">git rebase -i HEAD~&lt;number_of_commits_in_branch&gt;</code></pre>
    </div>
    <p>
      Squashing commits is a form of rebasing. The
      <code>
        -i
      </code>
      switch tells git you want to rebase interactively.
      <code>
        HEAD~&lt;number_of_commits_in_branch
      </code>
      indicates how many commits to look at for the rebase.
    </p>
    <p>
      Output is similar to:
    </p>
    <div class="highlight">
      <pre style="background-color:#f8f8f8;-moz-tab-size:4;-o-tab-size:4;tab-size:4"><code class="language-bash" data-lang="bash">pick d875112ca Original commit
pick 4fa167b80 Address feedback <span style="color:#666">1</span>
pick 7d54e15ee Address feedback <span style="color:#666">2</span>

<span style="color:#080;font-style:italic"># Rebase 3d18sf680..7d54e15ee onto 3d183f680 (3 commands)</span>

...

<span style="color:#080;font-style:italic"># These lines can be re-ordered; they are executed from top to bottom.</span></code></pre>
    </div>
    <p>
      The first section of the output lists the commits in the rebase. The second section lists the options for each commit. Changing the word
      <code>
        pick
      </code>
      changes the status of the commit once the rebase is complete.
    </p>
    <p>
      For the purposes of rebasing, focus on
      <code>
        squash
      </code>
      and
      <code>
        pick
      </code>
      .
    </p>
    <blockquote class="note">
  </li>
</ol>
<div>
  <strong>
    Note:
  </strong>
  For more information, see
  <a href="https://git-scm.com/docs/git-rebase#_interactive_mode">
    Interactive Mode
  </a>
  .
</div>
<ol start="2">
  <li>
    <p>
      Start editing the file.
    </p>
    <p>
      Change the original text:
    </p>
    <div class="highlight">
      <pre style="background-color:#f8f8f8;-moz-tab-size:4;-o-tab-size:4;tab-size:4"><code class="language-bash" data-lang="bash">pick d875112ca Original commit
pick 4fa167b80 Address feedback <span style="color:#666">1</span>
pick 7d54e15ee Address feedback <span style="color:#666">2</span></code></pre>
    </div>
    <p>
      To:
    </p>
    <div class="highlight">
      <pre style="background-color:#f8f8f8;-moz-tab-size:4;-o-tab-size:4;tab-size:4"><code class="language-bash" data-lang="bash">pick d875112ca Original commit
squash 4fa167b80 Address feedback <span style="color:#666">1</span>
squash 7d54e15ee Address feedback <span style="color:#666">2</span></code></pre>
    </div>
    <p>
      This squashes commits
      <code>
        4fa167b80 Address feedback 1
      </code>
      and
      <code>
        7d54e15ee Address feedback 2
      </code>
      into
      <code>
        d875112ca Original commit
      </code>
      , leaving only
      <code>
        d875112ca Original commit
      </code>
      as a part of the timeline.
    </p>
  </li>
  <li>
    <p>
      Save and exit your file.
    </p>
  </li>
  <li>
    <p>
      Push your squashed commit:
    </p>
    <div class="highlight">
      <pre style="background-color:#f8f8f8;-moz-tab-size:4;-o-tab-size:4;tab-size:4"><code class="language-bash" data-lang="bash">git push --force-with-lease origin &lt;branch_name&gt;</code></pre>
    </div>
  </li>
</ol>
<h2 id="contribute-to-other-repos">
  Contribute to other repos
</h2>
<p>
  The
  <a href="https://github.com/kubernetes">
    Kubernetes project
  </a>
  contains 50+ repositories. Many of these repositories contain documentation: user-facing help text, error messages, API references or code comments.
</p>
<p>
  If you see text you&rsquo;d like to improve, use GitHub to search all repositories in the Kubernetes organization.
  This can help you figure out where to submit your issue or PR.
</p>
<p>
  Each repository has its own processes and procedures. Before you file an
  issue or submit a PR, read that repository&rsquo;s
  <code>
    README.md
  </code>
  ,
  <code>
    CONTRIBUTING.md
  </code>
  , and
  <code>
    code-of-conduct.md
  </code>
  , if they exist.
</p>
<p>
  Most repositories use issue and PR templates. Have a look through some open
  issues and PRs to get a feel for that team&rsquo;s processes. Make sure to fill out
  the templates with as much detail as possible when you file issues or PRs.
</p>

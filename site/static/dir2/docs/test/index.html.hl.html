    <h2 id="lists">
      Lists
    </h2>
    <p>
      Markdown doesn&rsquo;t have strict rules about how to process lists. When we moved
      from Jekyll to Hugo, we broke some lists. To fix them, keep the following in
      mind:
    </p>
    <ul>
      <li>
        <p>
          Make sure you indent sub-list items
          <strong>
            4 spaces
          </strong>
          rather than the 2 that you
          may be used to. Counter-intuitively, you need to indent block-level content
          within a list item an extra 4 spaces too.
        </p>
      </li>
      <li>
        <p>
          To end a list and start another, you need a HTML comment block on a new line
          between the lists, flush with the left-hand border. The first list won&rsquo;t end
          otherwise, no matter how many blank lines you put between it and the second.
        </p>
      </li>
    </ul>
    <h3 id="bullet-lists">
      Bullet lists
    </h3>
    <ul>
      <li>
        This is a list item
      </li>
    </ul>
    <ul>
      <li>
        This is another list item in the same list
      </li>
    </ul>
    <ul>
      <li>
        You can mix
        <code>
          -
        </code>
        and
        <code>
          *
        </code>
        <ul>
          <li>
            To make a sub-item, indent two tabstops (4 spaces).
            <strong>
              This is different
              from Jekyll and Kramdown.
            </strong>
            <ul>
              <li>
                This is a sub-sub-item. Indent two more tabstops (4 more spaces).
              </li>
            </ul>
          </li>
          <li>
            Another sub-item.
          </li>
        </ul>
      </li>
    </ul>
    <!-- separate lists -->
    <ul>
      <li>
        <p>
          This is a new list. With Hugo, you need to use a HTML comment to separate two
          consecutive lists.
          <strong>
            The HTML comment needs to be at the left margin.
          </strong>
        </p>
      </li>
      <li>
        <p>
          Bullet lists can have paragraphs or block elements within them.
        </p>
        <pre><code>Indent the content to be one tab stop beyond the text of the bullet
point. **This paragraph and the code block line up with the second `l` in
`Bullet` above.**

```bash
ls -l
```

- And a sub-list after some block-level content</code></pre>
      </li>
      <li>
        <p>
          A bullet list item can contain a numbered list.
        </p>
        <ol>
          <li>
            Numbered sub-list item 1
          </li>
          <li>
            Numbered sub-list item 2
          </li>
        </ol>
      </li>
    </ul>

    <h2 id="code-blocks">
      Code blocks
    </h2>
    <p>
      You can create code blocks two different ways by surrounding the code block with
      three back-tick characters on lines before and after the code block.
      <strong>
        Only use
        back-ticks (code fences) for code blocks.
      </strong>
      This allows you to specify the
      language of the enclosed code, which enables syntax highlighting. It is also more
      predictable than using indentation.
    </p>
    <blockquote class="warning">
      <div>
        <strong>
          Warning:
        </strong>
        There is one situation where you need to use indentation for code blocks: when
        the contents of the code block contain lines starting with
        <code>
          -
        </code>
        or
        <code>
          *
        </code>
        characters.
        This is due to
        <a href="https://github.com/russross/blackfriday/issues/239">
          blackfriday issue #239
        </a>
        .
      </div>
    </blockquote>
    <pre><code>this is a code block created by back-ticks</code></pre>
    <p>
      The back-tick method has some advantages.
    </p>
    <ul>
      <li>
        It works nearly every time
      </li>
      <li>
        It is more compact when viewing the source code.
      </li>
      <li>
        It allows you to specify what language the code block is in, for syntax
        highlighting.
      </li>
      <li>
        It has a definite ending. Sometimes, the indentation method breaks with
        languages where spacing is significant, like Python or YAML.
      </li>
    </ul>
    <p>
      To specify the language for the code block, put it directly after the first
      grouping of back-ticks:
    </p>
    <div class="highlight">
      <pre style="background-color:#f8f8f8;-moz-tab-size:4;-o-tab-size:4;tab-size:4"><code class="language-bash" data-lang="bash">ls -l</code></pre>
    </div>
    <p>
      Common languages used in Kubernetes documentation code blocks include:
    </p>
    <ul>
      <li>
        <code>
          bash
        </code>
        /
        <code>
          shell
        </code>
        (both work the same)
      </li>
      <li>
        <code>
          go
        </code>
      </li>
      <li>
        <code>
          json
        </code>
      </li>
      <li>
        <code>
          yaml
        </code>
      </li>
      <li>
        <code>
          xml
        </code>
      </li>
      <li>
        <code>
          none
        </code>
        (disables syntax highlighting for the block)
      </li>
    </ul>
    <h3 id="code-blocks-containing-hugo-shortcodes">
      Code blocks containing Hugo shortcodes

    <p>
      To show raw Hugo shortcodes as in the above example and prevent Hugo

      <code>
        &gt;

      characters. The following example illustrates this (view the Markdown

    </p>

    <h2 id="images">
      Images
    </h2>
    <p>
      To format an image, use similar syntax to
      <a href="#links">
        links
      </a>
      , but add a leading
      <code>
        !
      </code>
      character. The square brackets contain the image&rsquo;s alt text. Try to always use
      alt text so that people using screen readers can get some benefit from the
      image.
    </p>
    <p>
      <img src="/static/images/pencil.png" alt="pencil icon">
    </p>
    <p>
      To specify extended attributes, such as width, title, caption, etc, use the
      <a href="https://gohugo.io/content-management/shortcodes/#figure">
        figure shortcode
      </a>
      ,
      which is preferred to using a HTML
      <code>
        &lt;img&gt;
      </code>
      tag. Also, if you need the image to
      also be a hyperlink, use the
      <code>
        link
      </code>
      attribute, rather than wrapping the whole
      figure in Markdown link syntax as shown below.
    </p>
    <figure>
      <img src="/static/images/pencil.png"
         alt="Image used to illustrate the figure shortcode" width="200px"/>
      <figcaption>
        <h4>
          Pencil icon
        </h4>
        <p>
          Image used to illustrate the figure shortcode
        </p>
      </figcaption>
    </figure>
    <p>
      Even if you choose not to use the figure shortcode, an image can also be a link. This
      time the pencil icon links to the Kubernetes website. Outer square brackets enclose
      the entire image tag, and the link target is in the parentheses at the end.
    </p>
    <p>
      <a href="https://kubernetes.io">
        <img src="/static/images/pencil.png" alt="pencil icon">
      </a>
    </p>
    <p>
      You can also use HTML for images, but it is not preferred.
    </p>
    <p>
      <img src="/static/images/pencil.png" alt="pencil icon" />
    </p>
    <h2 id="tables">
      Tables
    </h2>
    <p>
      Simple tables have one row per line, and columns are separated by
      <code>
        |
      </code>
      characters. The header is separated from the body by cells containing nothing
      but at least three
      <code>
        -
      </code>
      characters. For ease of maintenance, try to keep all the
      cell separators even, even if you heed to use extra space.
    </p>
    <table>
      <thead>
        <tr>
          <th>
            Heading cell 1
          </th>
          <th>
            Heading cell 2
          </th>
        </tr>
      </thead>
      <tbody>
        <tr>
          <td>
            Body cell 1
          </td>
          <td>
            Body cell 2
          </td>
        </tr>
      </tbody>
    </table>
    <p>
      The header is optional. Any text separated by
      <code>
        |
      </code>
      will render as a table.
    </p>
    <p>
      Markdown tables have a hard time with block-level elements within cells, such as
      list items, code blocks, or multiple paragraphs. For complex or very wide
      tables, use HTML instead.
    </p>
    <table>
      <thead>
        <tr>
          <th>
            Heading cell 1
          </th>
          <th>
            Heading cell 2
          </th>
        </tr>
      </thead>
      <tbody>
        <tr>
          <td>
            Body cell 1
          </td>
          <td>
            Body cell 2
          </td>
        </tr>
      </tbody>
    </table>
    <h2 id="sidebars-and-admonitions">
      Sidebars and Admonitions
    </h2>
    <p>
      Sidebars and admonitions provide ways to add visual importance to text. Use
      them sparingly.
    </p>
    <h3 id="sidebars">
      Sidebars
    </h3>
    <p>
      A sidebar offsets text visually, but without the visual prominence of
      <a href="#admonitions">
        admonitions
      </a>
      .
    </p>
    <blockquote>
      <p>
        This is a sidebar.
      </p>
      <p>
        You can have paragraphs and block-level elements within a sidebar.
      </p>
      <p>
        You can even have code blocks.
      </p>
      <div class="highlight">
        <pre style="background-color:#f8f8f8;-moz-tab-size:4;-o-tab-size:4;tab-size:4"><code class="language-bash" data-lang="bash">sudo dmesg</code></pre>
      </div>
    </blockquote>
    <h3 id="admonitions">
      Admonitions
    </h3>
    <p>
      Admonitions (notes, warnings, etc) use Hugo shortcodes.
    </p>
    <blockquote class="note">
      <div>
        <strong>
          Note:
        </strong>
        <p>
          Notes catch the reader&rsquo;s attention without a sense of urgency.
        </p>
        <p>
          You can have multiple paragraphs and block-level elements inside an admonition.
        </p>
        <p>
          | Or | a | table |
        </p>
      </div>
    </blockquote>
    <blockquote class="caution">
      <div>
        <strong>
          Caution:
        </strong>
        The reader should proceed with caution.
      </div>
    </blockquote>
    <blockquote class="warning">
      <div>
        <strong>
          Warning:
        </strong>
        Warnings point out something that could cause harm if ignored.
      </div>
    </blockquote>

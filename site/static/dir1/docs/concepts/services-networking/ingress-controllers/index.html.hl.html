    <h1>
      Ingress Controllers
    </h1>
    <p>
      In order for the Ingress resource to work, the cluster must have an ingress controller running.
    </p>
    <p>
      Unlike other types of controllers which run as part of the
      <code>
        kube-controller-manager
      </code>
      binary, Ingress controllers
      are not started automatically with a cluster. Use this page to choose the ingress controller implementation
      that best fits your cluster.
    </p>
    <p>
      Kubernetes as a project currently supports and maintains
      <a href="https://git.k8s.io/ingress-gce/README.md">
        GCE
      </a>
      and
      <a href="https://git.k8s.io/ingress-nginx/README.md">
        nginx
      </a>
      controllers.
    </p>

    <h2 id="using-multiple-ingress-controllers">
      Using multiple Ingress controllers
    </h2>
    <p>
      You may deploy
      <a href="https://git.k8s.io/ingress-nginx/docs/user-guide/multiple-ingress.md#multiple-ingress-controllers">
        any number of ingress controllers
      </a>
      within a cluster. When you create an ingress, you should annotate each ingress with the appropriate
      <a href="https://git.k8s.io/ingress-gce/docs/faq/README.md#how-do-i-run-multiple-ingress-controllers-in-the-same-cluster">
        <code>
          ingress.class
        </code>
      </a>
      to indicate which ingress controller should be used if more than one exists within your cluster.
    </p>
    <p>
      If you do not define a class, your cloud provider may use a default ingress controller.
    </p>
    <p>
      Ideally, all ingress controllers should fulfill this specification, but the various ingress
      controllers operate slightly differently.
    </p>

      <div>
        <strong>
          Note:

        Make sure you review your ingress controller&rsquo;s documentation to understand the caveats of choosing it.
      </div>
